1 · What it is

A full-stack application that converts a natural-language question into a safe, schema-aware SQL query, runs it against a user-registered PostgreSQL or MySQL database, and streams progress + results back to a React UI. The default inference path is a local Ollama model so nothing leaves the user's machine; cloud providers (OpenAI, Google Gemini, Groq, HuggingFace) can be selected per-request.

The unique value lives in the schema-retrieval RAG pipeline: instead of dumping the whole database schema into the prompt, the engine ranks tables/columns/join-paths against the user's question (BM25 + vector embeddings, fused with Reciprocal Rank Fusion), expands seeds by foreign-key neighbourhood, prunes irrelevant columns, and emits a compact CREATE TABLE + sample-rows context block sized for the model.

Local-first AI

Default Ollama with SQLCoder-8b. Privacy-friendly, free, works offline. Cloud providers are opt-in per query, keys passed inline (never stored server-side).

Multi-DB fan-out

Generate one SQL statement, execute it in parallel across multiple same-typed databases, and merge the rows with a synthetic __source_db__ column.

Streaming UX

Server-Sent Events report each pipeline stage (connect → schema → AI → validate → execute) so the UI shows live progress, not a spinner.

3 · High-level architecture

Three deployable processes when running in dev: the React/Vite dev server on port 3000 (proxying /api → backend), the FastAPI/Uvicorn backend on 8000, and the Ollama daemon on 11434. The backend orchestrates everything; the frontend never speaks directly to Ollama or to the user's databases.

Why the proxy? Vite is configured to forward every /api/* request to the backend, so the SPA can use relative URLs and avoid CORS friction. In production the same SPA can be served as static assets behind any reverse proxy.

4 · End-to-end query pipeline

A single natural-language query travels through five stages. Each stage emits an SSE progress event so the UI can render the live indicator; failure at any stage emits an SSE error event and stops the stream.

STAGE 01
Connect

Resolve target db_id(s), acquire async connection from pool, validate registration.

STAGE 02
Schema

Hybrid RAG: retrieve only the tables/columns relevant to the question; cache by (db_id, question-hash).

STAGE 03
AI

Build prompt (schema + intent rules) → call selected provider → extract SQL + explanation from response.

STAGE 04
Validate

sqlparse statement count, operation allow-list, injection pattern guard, auto-LIMIT on SELECT.

STAGE 05
Execute

Run under asyncio.wait_for timeout, normalise types (Decimal, datetime, bytes) for JSON, paginate.

4.1 — Single-DB stream (default path)

  1. UI POSTs the question + options to /api/v1/query/natural/stream?database_id=… and switches to text/event-stream reader mode.
  2. Backend opens an async connection via DatabaseConnectionManager, emits connect/completed.
  3. SchemaInspector.get_relevant_schema_summary runs the hybrid retriever (see §5). Result is a CREATE-TABLE-style text block including foreign keys and 1 sample row per table.
  4. QueryIntentDetector classifies the question (single-read, multi-step, aggregation, complex-filter, write…) so the prompt builder can splice in intent-specific guidance.
  5. prompts.build_sql_generation_prompt assembles the final prompt that ends with the seeded ```sql opener — the Ollama stop-token cuts after the next closing fence.
  6. generate_with_config routes to the configured provider (Ollama/OpenAI/Gemini/Groq) with retry-on-timeout for Ollama.
  7. prompts.extract_sql_from_response handles raw SQL, fenced blocks, JSON wrappers, or prose-prefixed forms. Explanation is either extracted or synthesised from the parsed SQL.
  8. QueryValidator.validate enforces statement limits, optional strict sanitiser checks, and auto-appends LIMIT on SELECT.
  9. If execute=true, QueryExecutor.execute runs the SQL, paginates results, returns rows + execution time. Otherwise the UI shows an editable SQL preview the user can run via /query/sql.
  10. Final result SSE event carries the full QueryResponse JSON; UI renders the table and appends a history entry.

4.2 — Multi-DB fan-out

When database_ids=id1,id2,id3 is passed, the backend takes the same five stages but:

  • Validates that all selected databases share the same engine type (mixed types are rejected — the generated SQL would not be portable).
  • Uses the first database's schema for prompting (the assumption: same schema family).
  • Executes the validated SQL on each DB in parallel via asyncio.gather; failures become warnings, partial success still produces a merged result set.
  • Prepends a synthetic __source_db__ column to merged rows so the UI can group by source.

4.3 — Raw SQL mode

The UI's "Raw SQL" toggle skips AI entirely and posts directly to /query/sql. Validation still runs; multi-DB fan-out from the UI is done client-side by issuing N parallel POSTs.

4.4 — Write operations

UPDATE/INSERT/DELETE are allowed by default. The validator permits a compound shape of write-then-SELECT in one request (useful for "delete X and show me what's left"). The UI detects write intent and shows a banner warning before the user clicks "Run anyway".

5 · Hybrid schema retrieval (RAG) — deep dive

This is the engine's most important subsystem. Treat the rest of the project as plumbing; the RAG pipeline is what makes generated SQL actually correct.

5.0 — The problem it solves

A real database has hundreds of tables and thousands of columns. Three things go wrong if you naively concatenate all of it into the model prompt:

  • Context window blow-out. SQLCoder-8b runs at roughly 4–8k tokens of usable context. A 200-table schema with sample rows easily exceeds that. The model silently truncates the schema, hallucinates the rest.
  • Attention dilution. Even within budget, the relevant tables get drowned in noise. The model picks plausible-looking table names rather than the right ones.
  • Latency. Every extra token costs decode time. A 6k-token prompt at 30 tok/s is a 3 minute wait.

The retriever's job: given a natural-language question and the database's full schema metadata, return a compact, focused text block containing only the tables and columns the question actually needs, plus the foreign keys that join them, plus 1 sample row each. The output goes verbatim into the AI prompt's ### Database Schema section.

5.1 — Why "hybrid"

Two retrieval signals catch different things, and neither alone is enough:

SignalCatchesMisses
BM25 (lexical)Direct word matches: "show customers" → the customers table.Synonyms, paraphrases, semantic intent. "Who pays us?" misses customers entirely.
Vector (semantic)Synonyms and concepts: "revenue" → orders.total, "client" → customers.Exact tokens the embedder hasn't seen, and rare proper nouns (column names like cust_xref_id_v3).

The hybrid pipeline runs both, fuses the rankings, then expands by the foreign-key graph so join-target tables are always present even if they didn't directly match either signal.

This is the engine's secret sauce. A real database can have hundreds of tables; dumping all of them blows the context window and degrades SQL quality. The pipeline below picks the right subgraph.

5.1 — Inputs

  • Question — the user's raw natural-language string.
  • tables_info — list of {name, columns:[{name,type,nullable,is_pk,is_fk,…}]} from _get_all_tables_info.
  • foreign_keys — list of {table, column, ref_table, ref_column} tuples.
  • schema_hash — MD5-derived integer of column metadata; identifies a unique schema version for on-disk index invalidation.

5.2 — Stage 0 · Schema introspection & corpus build

Before retrieval can run, the engine needs structured metadata. SchemaInspector._get_all_tables_info issues a single information_schema.columns query, then enriches each column with PK and FK flags from two more queries. A separate query pulls every FK edge as {table, column, ref_table, ref_column}. A schema-version hash (MD5 of concatenated table+column+type strings) is computed — this single integer identifies a unique schema state and is used to invalidate the on-disk vector index when the schema changes.

The SchemaCorpusBuilder converts the raw metadata into three deterministic document lists — these are the units that get ranked or embedded:

Doc kindKeyText contentUsed by
Table doctable_name"{name}. columns: {col1, col2, …}. types: {type1, type2, …}"BM25 ranker + table vector search
Column doc(table, column)"{col_name} ({col_type}) in table {tbl}"Column pruner (cosine scoring per column)
Path doc(tableA, tableB)"join {A} and {B} on {colA}={colB}. tableA columns: …. tableB columns: …"Path vector search — semantic join discovery

5.3 — Stage A · BM25 lexical ranking

Pure-Python BM25 over table documents only (column-level lexical is handled implicitly because the table doc embeds its column names with weight ×2).

Tokenization (deterministic):

  • Regex split on [a-zA-Z][a-zA-Z0-9]* — drops punctuation and pure-numeric noise.
  • Lower-case.
  • Split on underscores so customer_orders contributes both customer and order.
  • Drop a 17-word stop list (show, list, the, give, with…) — these are user-instruction words, not schema content.
  • Naive plural stemming: -ies → -y, trailing -s stripped. "orders" and "order" land in the same bucket.

Weighted doc construction: table-name tokens are emitted ×3, column-name tokens ×2, column-type tokens ×1. So a table whose name matches the query outranks one where only a column name matches — which is exactly what you want.

The BM25 formula (constants k₁ = 1.5, b = 0.75):

score(D, Q) = Σt∈Q IDF(t) · ( f(t,D) · (k₁+1) ) / ( f(t,D) + k₁ · (1 − b + b · |D| / avgdl) )

IDF(t) = log( 1 + (N − n(t) + 0.5) / (n(t) + 0.5) )

where f(t,D) is term frequency in the table-doc, |D| its token length, avgdl the corpus average, N the number of tables, n(t) the number of tables containing the term.

Why BM25 not pure TF-IDF: the b · |D|/avgdl term penalises long table-docs (wide tables with many columns) so a small table with an exact name match still wins against a sprawling table that mentions the term incidentally. The (k₁+1) · f / (f + k₁·…) shape saturates: a column-name appearing 10× doesn't dominate a column-name appearing 3×.

Output: a list of TableScore(name, score) sorted descending. Only scores > 0 are kept as the BM25 ranking that feeds RRF.

5.4 — Stage B · Vector index (semantic retrieval)

Built only when sentence-transformers is installed and HYBRID_RETRIEVAL_ENABLED=true. If the library is missing, the embedder raises a sentinel exception and the pipeline transparently degrades to BM25-only — there is no user-visible failure.

5.4.1 — The embedder

SchemaEmbedder wraps a sentence-transformer model (typically a small all-MiniLM-class encoder, ~22M params, 384-dim output) loaded lazily on first use. Outputs are L2-normalised so cosine similarity reduces to a dot product — that makes the search code branch-free and fast.

5.4.2 — Three parallel vector spaces

The corpus's three doc lists become three NumPy matrices in a single VectorIndex:

  • table_vecs — one row per table. Cosine-matched against the question for top-20 candidate tables.
  • column_vecs — one row per column. Not used in the initial table selection (would be too noisy at the table-set level), but is the engine behind column pruning (§5.7) once tables are chosen.
  • path_vecs — one row per FK edge, described in human-readable English ("join orders and customers on customer_id=id"). Top-10 paths are retrieved, then both endpoints are flattened into a table candidate list. This is how semantic join discovery works: if you ask "what did our biggest spenders buy?", the path orders ↔ customers can score high even when only "spend" matched orders.total.

5.4.3 — On-disk persistence + invalidation

Indexes are saved as .npz under ~/.nlsql/embeddings/ with the key:

{db_id}__{schema_hash}__{model_name_slug}.npz
  • schema_hash in the filename means a DDL change auto-invalidates the index — next request rebuilds, no manual cache bust.
  • model_name in the filename means switching the embedder rebuilds the index rather than mixing dimensions.
  • A per-db_id asyncio.Lock prevents two concurrent first-time requests from racing into a parallel rebuild.
  • Cleanup keeps the 3 newest indexes per database; older files are swept on each new build.
  • Corrupted files (failed np.load) are deleted and rebuilt rather than fatal.

5.4.4 — Search

At query time the question is embedded once (a single forward pass through the encoder) and the resulting 384-d vector is dot-producted against table_vecs and path_vecs. np.argpartition picks the top-k without a full sort. Output of this stage: a "table-rank" list and a "path-expanded" list — both feed RRF next.

5.5 — Stage C · Reciprocal Rank Fusion

Three ranked lists now exist:

  1. BM25 tables — lexical signal.
  2. Vector tables — semantic signal at the table level.
  3. Path-expanded tables — semantic signal at the relationship level, flattened to endpoint tables.

Fusing them naïvely (e.g. averaging scores) fails because the scores live on incomparable scales — BM25 outputs sometimes-double-digit floats; cosine sits in [-1, 1]. RRF normalises by rank position alone and ignores raw scores:

RRFscore(item) = Σi ∈ rankings 1 / (k + ranki(item))
k = 60 (default — the published RRF constant from Cormack et al. 2009).

Properties that matter here:

  • Robust to outlier scorers. One signal returning a wild score can't capsize the fusion.
  • Stable ordering. The implementation breaks ties by first-appearance order across the input rankings — important so identical schemas yield identical contexts (which is important for caching).
  • Cheap. Pure-Python dict accumulation; O(total ranks).

Output: a single fused list of table names in priority order. The top MAX_SEED_TABLES (default 5) become "seeds" for the next stage.

5.6 — Stage D · Foreign-key expansion

Seeds alone are not enough: a question like "customers' total spend last month" wins seeds customers and orders, but a correct query also needs the order_items table for line-item totals. Two pieces of intuition drive this stage:

  • The SQL the model is about to write almost always involves joins along foreign keys.
  • If a related table is missing from the prompt, the model will either hallucinate column names or fall back to flat queries that ignore the join.

So the expander builds an undirected adjacency map from the FK list (an edge is added only when both endpoints are present in the schema) and walks 1-hop from each seed in order. For each seed it appends the seed itself (if new) and then its sorted neighbour set (each skipped if already seen). This is bounded — the final table list is truncated at MAX_TABLES (default 12).

Why 1-hop and not BFS to depth 2? Empirically, 2-hop blows the budget on schemas with hub tables (e.g. users connected to 30 other tables). 1-hop with a small budget is the right trade-off — if a deeper join is actually needed, the question's vocabulary usually elevates the intermediate table on its own through BM25 or vector matching.

5.7 — Stage E · Column pruner

Some tables are wide — 80+ columns is common in legacy enterprise schemas. Including all of them spends prompt budget on irrelevant columns and confuses the model. The pruner shrinks wide tables to a budget (MAX_COLS_PER_TABLE = 15 by default) without losing structurally necessary columns. Tables with fewer columns than the budget are short-circuited and pass through unchanged.

The pruning algorithm computes a cosine score per column (column_vec · question_vec) and then keeps a union of three sets:

  1. Mandatory — every column with is_pk = True or is_fk = True. Without these the model can't join or identify rows, no matter how irrelevant their names look. This is a structural constraint, not a relevance one.
  2. Top-N by score — sorted by cosine, fill remaining budget slots (budget − |mandatory|). These are the "this column probably contains the answer" picks.
  3. Above-threshold — any column scoring > COLUMN_SCORE_THRESHOLD (0.25) is kept even if it overruns the budget. Rationale: when the user's question is unambiguous about which columns matter, the budget should yield.

The pruner preserves the columns' original schema order — important because models that emit DDL-shaped responses tend to mirror the order they saw the columns in.

5.8 — Stage F · Sample-row hydration

For each surviving table the engine fetches a small sample (default 1 row) via SELECT * FROM "{table}" LIMIT 1. The table name is validated against an identifier regex before interpolation — no user input is involved here, but the safety belt stays on. Sample values are coerced to JSON-safe types (Decimals → str, datetimes → str, bytes → str).

Why sample rows matter:

  • Type disambiguation. The column type says VARCHAR(50); the sample row tells the model whether that's an ISO date string or a free-text label.
  • Encoding clues. A status column whose sample reads 'A' hints at single-letter codes; 'active' hints at long-form enums.
  • Foreign-key conventions. The sample shows whether customer_id looks like an integer or a UUID — affects the SQL the model writes.

Failures (permission denied on a particular table, broken DDL) are logged and silently skipped — a missing sample is better than aborting the whole pipeline.

5.9 — Stage G · Format the context block

The pure format_schema_context function renders the final text. The output mimics what a database administrator would paste into a help thread — a shape SQL-tuned models speak natively:

  • Header comment listing the available table names with an emphatic "use only these exact names, do not guess" — empirically reduces table-name hallucination.
  • One CREATE TABLE block per selected table, with NOT NULL and DEFAULT annotations and inline FOREIGN KEY declarations only when the referenced table is also in the selected set (so the model never sees a dangling reference).
  • A -- Sample rows from {table}: block under each CREATE TABLE with up to N rows printed as comma-separated col=value pairs.
  • A -- RELATIONSHIPS footer listing every FK edge between selected tables in A.col → B.col form. The model uses this as a fast lookup for joins it should consider.
  • If BM25 returned zero matches (very generic question), a fallback notice is prepended and the first N tables are emitted — keeps the engine from producing an empty schema in pathological cases.

5.11 — Progress events

Each stage emits a ProgressEvent through an optional emitter callable. The streaming endpoint subscribes and forwards them as SSE so the UI can show "BM25 ✓ (12ms) · Embed cached · Vector ✓ (8ms) · Fuse ✓ (1ms) · FK ✓ · Prune ✓ (3ms) · Samples ✓ (45ms)" in real time. Stages legitimately skipped (vector path when BM25-only) emit status: skipped rather than disappearing — important UX signal.

5.13 — Why this design beats the alternatives

AlternativeWhy this project doesn't use it
Dump full schema into promptBreaks on real schemas; degrades SQL quality even when it fits.
Vector-only retrievalMisses exact column-name matches; embedders trained on prose underperform on identifier strings.
BM25-only retrievalMisses paraphrases ("clients" vs "customers"); no help when the question's vocabulary doesn't overlap with schema names.
LLM-as-retriever (ask the model which tables to use)Adds a full second model round-trip; needs the schema in that prompt too — chicken and egg.
Pre-computed Q→table mappingsDoesn't generalise; the system would need labelled data per database.
Score-weighted fusion of BM25 + cosineRequires score normalisation across incomparable scales. RRF sidesteps this by using rank-only.
Graceful degradation. If embeddings library is missing or fails to load, the retriever drops to BM25-only and still produces a valid context. If BM25 finds nothing (highly generic question), a bounded fallback (first N tables) is used with an explicit comment in the context.

5.8 — Caching layers

  • In-memory TTLCache (cachetools) keyed by (db_id, max_tables, md5(question)), 1-hour TTL by default.
  • On-disk vector index (.npz), keyed by schema hash and model. Up to 3 newest indexes kept per database; older ones swept.
  • Schema-version hash recomputed each call to detect DDL changes and invalidate stale indexes.

7.1 — End-to-end UX flow

  1. User lands on http://localhost:3000 → app loads database list (GET /databases) and polls it every 30s.
  2. User clicks "Connect new" in sidebar → ConnectionForm opens. They pick a type (PG / MySQL), defaults populate, optionally hit "Test" to validate live, then "Save".
  3. User multi-selects one or more databases via the sidebar checkboxes. Selected DBs render as chips above the query box.
  4. In AI mode, they type a question. Hit ⌘↵ or "Generate SQL".
  5. Frontend opens an SSE stream to /query/natural/stream. QueryProgress animates stage-by-stage.
  6. If "Preview SQL" was pressed instead, the stream stops after stage 4 (validate) and renders an editable SQL box. User can edit and click "Run This Query".
  7. Results land in ResultsDisplay. The query is appended to history. Errors surface as toasts and inline banners.
  8. Settings modal lets the user switch provider (Ollama default, OpenAI / Gemini / Groq with an inline API key field). The choice is per-session and rides with every request.

10 · Security model

This is a developer-facing analytics tool. It executes user-provided SQL against user-provided databases. Treat the backend as trusted only to the extent the operator trusts the queries they're sending it.
  • Credential storage. Saved DB passwords are Fernet-encrypted in ~/.nlsql/databases.json. If DB_ENCRYPTION_KEY is empty, a temporary key is generated per process — saved passwords cannot be decrypted after restart. Always set a stable key in production.
  • Statement limits. The validator allows one statement, or one write statement followed by one SELECT. Multi-statement attacks are rejected.
  • Operation allow-list. SELECT, INSERT, UPDATE, DELETE, WITH (CTE), EXPLAIN. DDL (CREATE/ALTER/DROP/TRUNCATE) is blocked unless explicitly allowed in code.
  • Pattern guard. The sanitiser regex-blocks dangerous functions and known UNION-injection signatures. Strict mode adds comment/hex-literal blocks.
  • Auto-LIMIT. Bare SELECTs get DEFAULT_QUERY_LIMIT appended; oversized explicit limits are capped at MAX_QUERY_RESULTS.
  • Query timeout. Every execution runs under asyncio.wait_for(QUERY_TIMEOUT_SECONDS); the connection is released on timeout.
  • Identifier validation. Schema-inspector validates table names against ^[a-zA-Z_][a-zA-Z0-9_]*$ before any quoted interpolation in sample-row fetches.
  • CORS. Defaults to localhost:3000; tighten in production.
  • API keys for cloud providers are passed inline per request, never logged, never stored. The server is a passthrough.
  • Recommended hardening. Use a read-only DB user for analytics workloads to make the write-allow setting irrelevant. Run behind an authenticating reverse proxy. Set a stable Fernet key. Lower API_RATE_LIMIT_PER_MINUTE if exposed.

11 · Run & install flow

11.1 — install.sh

One-shot bootstrapper. Steps:

  1. Detects OS + CPU arch (macOS/Linux/Windows-WSL, x86_64/ARM64).
  2. Checks Docker, Python 3.12+ (prefers uv if present), Node 18+, npm. Prints platform-specific install commands if anything is missing.
  3. Creates .venv, installs requirements.txt, marks the venv as ready.
  4. npm install inside frontend/.
  5. Copies .env.example → .env if absent.
  6. Spins up a Docker-Compose Ollama service (optional, if docker-compose.yml exists) and pulls a starter model.
  7. Creates ~/.nlsql/ with mode 700 and a local logs/ directory.

11.2 — run.sh

Process orchestrator with subcommands:

  • dev — ensure Ollama is up (launchctl / brew services / spawn), confirm the model is pulled (auto-pull otherwise), warm the model with a tiny generation call, start uvicorn with --reload, start Vite, monitor both PIDs, tear everything down on SIGINT.
  • prod — backend only, no reload, load .env into the environment.
  • setup-ollama — install Ollama (via brew) if missing, then run the ensure-and-warm sequence.
  • stop — kill anything bound to ports 3000 and 8000.
  • clean — wipe logs, __pycache__, frontend/.vite.
  • logs — interactive picker for backend / frontend / combined tail.

11.3 — Seed scripts

scripts/seed_postgres.sh and scripts/seed_mysql.sh spin up a sample schema for local testing. scripts/migrate_encrypt_passwords.py retrofits encryption onto an older plaintext databases.json.

13 · How to rebuild from scratch

A precise, ordered checklist that, combined with the structure above, lets a new engineer reconstruct the project end-to-end:

  1. Scaffold backend. FastAPI app with a v1 router prefix, Pydantic Settings loading .env, async lifespan that verifies Ollama and optionally registers a DB from env. CORS + global NLSQLException handler. Prometheus counters + /metrics.
  2. Define the exception hierarchy in app/exceptions.py — every layer raises typed errors; only the global handler converts them to HTTP responses.
  3. Define request/response models in app/models/ — connections, queries, options, pagination, execution result, query plan.
  4. Build the database adapter layer. Abstract base + PG / MySQL implementations for connection URL, schema name, size + connection-count queries.
  5. Implement the connection manager. Per-database async engine pool, Fernet-encrypted persistence to ~/.nlsql/databases.json, registration/disconnect lifecycle, async context manager for connections.
  6. Implement schema introspection. Pull tables/columns/PK/FK from information_schema, compute a schema-version hash, expose two summary calls (full and RAG-relevant).
  7. Build the RAG pipeline in this order: corpus builder → BM25 ranker → vector embedder wrapper → vector index (save/load to .npz) → RRF fuse → FK expander → column pruner → formatter → HybridRetriever orchestrator. Add a tiny progress emitter so the streaming endpoint can observe sub-stages.
  8. Build the AI layer. Prompt template with intent-aware blocks, SQL/explanation extractors with multiple fallback parsing strategies, intent detector with the nine intent classes, Ollama HTTP client with retry/backoff, multi-provider router for OpenAI/Gemini/Groq/HF, SQLGenerator orchestrator that wires intent + schema + prompt + provider.
  9. Build the query layer. Validator (sqlparse statement count, operation allow-list, auto-LIMIT, optional strict sanitiser), executor (asyncio.wait_for timeout, pagination, type coercion), error humaniser.
  10. Wire endpoints. Five endpoint files (health, query, query_stream, schema, database) all attached under /api/v1 via a single router aggregator. The streaming endpoint is the most complex — it owns the SSE event format and the multi-DB fan-out path.
  11. Scaffold the React app. Vite + Tailwind 4 + Radix. Build the layout shell (top nav, sidebar, main, settings modal), then the connection form, then the query interface (with custom SSE reader), then the results display.
  12. Connect the dev proxy. Vite proxies /api to :8000 so the SPA can use relative URLs.
  13. Write install.sh + run.sh for one-command bootstrap and lifecycle, including Ollama detection, model pull, warm-up generation, and orderly shutdown.
  14. Ship .env.example with every variable documented inline. Validate at startup via Pydantic field validators.
  15. Add tests under tests/: sanitiser patterns, validator behaviour, prompt builder, SQL extractor, RAG retrieval components (BM25, RRF, FK expander, column pruner, corpus, vector index, formatter), schema inspector hybrid path, SQL generator hybrid path, query-stream SSE shape, exceptions, config validators, error humaniser.
  16. Last: wire Prometheus counters/histograms in the streaming endpoint and the AI client, then expose /metrics. structlog for everything else.
The non-obvious design decisions that make the project tick: (1) the RAG context is rendered as CREATE TABLE + sample-row comments — SQL models speak this dialect natively; (2) the prompt is seeded with a trailing ```sql opener so Ollama emits raw SQL with no prose; (3) intent detection lives before retrieval so the prompt block can be tuned to the question shape; (4) the vector index is keyed by schema-hash, so DDL changes auto-invalidate without manual cache busts; (5) write-then-SELECT is a permitted compound shape so "delete X and show me what's left" works in one round-trip.
